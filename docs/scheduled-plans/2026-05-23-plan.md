## Slice plan

### Slice 1 — Penyelarasan Rute Surfaces & Registry
- Goal: Menyelesaikan 11 *palette drift* dan 7 *route drift* untuk memastikan `vac doctor surfaces` lulus dengan sukses (exit code 0).
- Why: **Root Cause** berupa ketidakselarasan rute. Manifest kapabilitas di `.vac/capabilities/` mengklaim rute permukaan (`palette`, `cli`, `tui`) yang tidak dideklarasikan di berkas surfaces terkait. Sebaliknya, terdapat rute permukaan (seperti rute visible `/debug-config` untuk `vac.sessions` pada `tui.yaml`) yang tidak terdaftar di bawah `surfaces` manifest kapabilitas.
- Files:
  - [tui.yaml](file:///.vac/surfaces/tui.yaml)
  - [cli.yaml](file:///.vac/surfaces/cli.yaml)
  - [palette.yaml](file:///.vac/surfaces/palette.yaml)
  - [slash.yaml](file:///.vac/surfaces/slash.yaml)
  - [capabilities/*.yaml](file:///.vac/capabilities/) (seperti `approvals.yaml`, `chat.yaml`, `sessions.yaml`, `tools.yaml`, `tui.yaml`, dll.)
- Steps:
  1. Audit semua rute yang dideklarasikan di surfaces. Tambahkan rute palet yang hilang pada `.vac/surfaces/palette.yaml` untuk kapabilitas seperti `vac.approvals`, `vac.architecture`, `vac.build`, `vac.donor_migration`, `vac.identity.check`, `vac.ownership`, `vac.release`, `vac.sandbox`, `vac.tools`, `vac.tui.pty_gate`, dan `vac.tui`.
  2. Tambahkan rute CLI yang hilang pada `.vac/surfaces/cli.yaml` untuk `vac.approvals` (`vac exec`, `vac review`), `vac.chat` (`vac exec`), `vac.sessions` (`vac`), `vac.tools` (`vac exec`).
  3. Tambahkan rute TUI/Slash yang hilang pada `.vac/surfaces/tui.yaml` dan `.vac/surfaces/slash.yaml` untuk kapabilitas `vac.tui` (seperti `/`, `/capabilities`, `/workflow`, `/debug-config`, `/activity`, `/approvals`, `/evidence`).
  4. Hapus rute permukaan yang tidak valid/tidak diakui manifest kapabilitas (seperti `/debug-config` untuk `vac.sessions`) atau daftarkan dengan benar di bawah `surfaces` manifest kapabilitas.
- Validation:
  - `./vac-rs/target/debug/vac doctor surfaces .`
  - `./vac-rs/target/debug/vac doctor registry .`
- Docs sync:
  - [CAPABILITY_MAP.md](file:///docs/product/CAPABILITY_MAP.md) dan [CAPABILITY_PRD_COVERAGE.md](file:///docs/product/CAPABILITY_PRD_COVERAGE.md) diperbarui jika terjadi pemetaan ulang permukaan fitur.
- Risks:
  - Risiko: Kesalahan pemetaan rute dapat merusak aksesibilitas menu navigasi visual TUI atau respons perintah CLI.
  - Rollback: `git checkout -- .vac/surfaces/*.yaml` untuk mengembalikan ke konfigurasi terakhir yang stabil.
- UX impact: Rute UI dan dasbor kapabilitas memetakan fitur secara presisi, menjamin transparansi operasional yang bersih bagi operator.
- Owner/Coordination: **Frontend Lead** & **Security Architect** untuk sinkronisasi rute permukaan dengan kontrol akses runtime.

### Slice 2 — Standardisasi Format Kepemilikan (Owner)
- Goal: Mengeliminasi 6 *owner conflicts* agar laporan diagnosa registry terbebas dari inkonsistensi kepemilikan modul.
- Why: **Root Cause** berupa ketidaksesuaian string nama pemilik kapabilitas utama di berbagai surfaces. Contoh: `vac.approvals` dideklarasikan sebagai `vac-rs/cli` di `cli.yaml` tetapi `vac-tui::approvals` di `slash.yaml` dan `vac-tui/bottom_pane.approval_overlay` di `approvals.yaml`.
- Files:
  - [capabilities/*.yaml](file:///.vac/capabilities/) (seperti `approvals.yaml`, `chat.yaml`, `identity.yaml`, `sessions.yaml`, `tools.yaml`, `workflow.yaml`)
  - [surfaces/*.yaml](file:///.vac/surfaces/)
- Steps:
  1. Identifikasi 6 kapabilitas berkonflik (`vac.approvals`, `vac.chat`, `vac.sessions`, `vac.workflow`, `vac.tools`, `vac.identity`) melalui log diagnosa.
  2. Normalisasikan string pemilik secara seragam. Samakan route owner pada file surfaces dengan capability owner utama di file manifest kapabilitas, atau gunakan format terstandar.
  3. Lakukan modifikasi pada bagian `owner` dan `ownership` agar selaras 100%.
- Validation:
  - `./vac-rs/target/debug/vac doctor registry .`
  - Unit test: `cargo test --manifest-path vac-rs/Cargo.toml -p vac-core registry_diagnostics -- --nocapture`
- Docs sync:
  - [CAPABILITY_MAP.md](file:///docs/product/CAPABILITY_MAP.md) jika ada pembaruan representasi kepemilikan kapabilitas.
- Risks:
  - Risiko: Kesalahan penamaan pemilik dapat memicu kegagalan sistem deteksi arsitektur/registry otomatis.
  - Rollback: `git checkout -- .vac/capabilities/*.yaml .vac/surfaces/*.yaml`
- UX impact: Kepemilikan kode menjadi sangat jelas, sehingga memudahkan pelacakan bug modular dan audit hak akses.
- Owner/Coordination: **Core Architecture Lead** untuk standarisasi format penamaan yang selaras dengan crate Rust.
