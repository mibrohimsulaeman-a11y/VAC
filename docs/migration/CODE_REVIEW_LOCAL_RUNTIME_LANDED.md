# Code review — landed Local Runtime Contract work (00B + 00C-default + 00D-5 + 00D-8)

**Date:** 2026-05-06
**Reviewer:** Notion AI (MCP session) wearing three hats: code reviewer, software architect, software engineer.
**Scope:** All code that has landed for ADR-0007 "Local Runtime Contract" Phase 0 sub-plans: 00B (contract types), 00C-default-prompt (vac-exec wiring), 00D-5 (TUI canonical input), 00D-8 (Activity Sidebar).
**Repo HEAD:** `35aaa5d` on `main`. `cargo check -p vac-surface-cli` passes (28.79s; only dead_code warnings in vac-tui).

This review supplements `00E_REACHABILITY_AUDIT.md` (which covered dependency reachability) and `00D_RESIDUAL_LEGACY_AUDIT.md` (which covered the unfinished migration scope). Here the focus is **the *quality* of what is already in main**: design decisions, abstractions, idioms, correctness, performance, testability.

Landings reviewed:

- `vac-rs/core/src/local_runtime/{mod,approval,command,error,event,ids,session,task,validation,tests}.rs` (~22 KB total).
- `vac-rs/exec/src/local_runtime.rs` (~1283 LoC).
- `vac-rs/exec/src/lib.rs` (1886 LoC) — reviewed for the runtime-first cutover, not the legacy resume/review paths.
- `vac-rs/tui/src/local_runtime.rs` (~113 KB; head + structural references via grep).
- `vac-rs/tui/src/activity_sidebar.rs` (~330 LoC, full).

## TL;DR

The contract is **structurally sound and product-faithful** — ADR-0007's terminology rule is honored, ID newtypes are consistent, and the projection seam (Event → RuntimeEvent) is in the right place. **What's weak is mostly engineering hygiene, not architecture**:

1. The `impl_display_as_str!` macro is duplicated four times. Consolidation is trivial and high-value.
2. `vac-rs/tui/src/local_runtime.rs` at ~113 KB / >2500 LoC in one file is a maintenance hazard.
3. `RuntimeSession`/`RuntimeTask` carry `status` fields that are mutated by direct field assignment from adapters — no FSM, no transition methods, no invariants.
4. Two near-identical event projectors live in `vac-exec` and `vac-tui`. Shared projection should be lifted into `vac-core`.
5. `LocalRuntimeHumanOutput` writes to `eprintln!`/`println!` directly — untestable without process boundaries.
6. There are no round-trip serde tests pinning the wire format.

Correctness-wise the only real bugs found are:

- `assistant_buffer` and `evidence` in `LocalRuntimeBridge` grow unbounded.
- Stale-turn events in the bridge are silently dropped (no debug log, no metric).
- `LegacyCompatTransport` is a single-variant enum guarded by `debug_assert_eq!` — a smell, but harmless today.

The rest is improvements, not fixes. Recommendations are prioritized at the end.

## A. `vac-rs/core/src/local_runtime/` — Contract types

### Strengths

- **Product-level naming everywhere.** No leakage of `Thread*`/`Client*`/`Turn*` into the contract; ADR-0007's terminology rule is enforced and visible.
- **Typed IDs via newtype-with-macro** (`define_runtime_id!`). Each ID type is `Copy + Eq + Hash + Serialize + Deserialize`, with `from_uuid`, `from_string`, `as_uuid`, `From<Uuid>`, `TryFrom<&str>`, `TryFrom<String>`, `FromStr`, `Display`. Comprehensive and prevents `TaskId` ↔ `SessionId` mix-ups.
- **Display/`as_str` helpers** via `impl_display_as_str!` macro give operator-friendly stringification with one-line variant maps.
- **Builder helpers** on every type (`StartTask::new`, `RuntimeCommand::start_task`, `TaskStarted::new`, `ToolCallStarted::new`, etc.) accept `impl Into<...>`. Ergonomic.
- **Tests exist** (8 focused tests) and cover construction + display + a happy-path approval roundtrip.
- **Serde with `#[serde(rename_all = "snake_case")]`** consistently — wire shapes are predictable.

### Issues

#### A.1 `impl_display_as_str!` macro is duplicated four times

Defined verbatim in `approval.rs`, `session.rs`, `task.rs`, and `validation.rs`. Same macro body. Each copy is ~14 lines.

**Fix.** Define once in `mod.rs` (or a private `macros.rs`) with `macro_rules!` + `pub(crate) use macros::impl_display_as_str;`. Reduces ~50 LoC of pure duplication and removes a four-way drift hazard.

#### A.2 `RuntimeSession::status` and `RuntimeTask::status` are unmanaged

Both structs expose `pub status: ...Status` fields. The exec adapter mutates them directly:

```rust
// vac-exec/src/local_runtime.rs (LocalRuntimeBridge::map_core_event)
let mut task = RuntimeTask::new(self.session_id, RuntimeTaskKind::SemanticCoding, self.task_prompt.clone());
task.id = self.task_id;                       // overrides freshly-minted ID
task.status = RuntimeTaskStatus::Running;     // direct field write
events.push(RuntimeEvent::TaskStarted(TaskStarted::new(task)));
```

This means:

- The constructor `RuntimeTask::new` always returns `status: Pending`, but adapters override it post-hoc — the constructor's invariant lies.
- There's no FSM. Nothing prevents `Cancelled → Running`.
- The `id` is regenerated by `TaskId::new()` and then thrown away — wasted allocation and `Uuid::new_v4()` call per event.

**Fix.** Add transition methods (`RuntimeTask::start_running`, `RuntimeTask::wait_approval`, etc.) and either:

- Make `id` and `status` private with getters, or
- Provide an explicit `RuntimeTask::with_id(session_id, id, kind, prompt)` constructor for adapters that already own the ID.

Same applies to `RuntimeSession::status` (currently never mutated anywhere — effectively a constant `Active`, which is misleading).

#### A.3 `RuntimeEvent::ApprovalRequested` carries a fat variant

```rust
#[allow(clippy::large_enum_variant)]
pub enum RuntimeEvent { /* ... */ ApprovalRequested(ApprovalRequest), /* ... */ }
```

`ApprovalRequest` is ~10 fields including `Vec<ApprovalResource>`, `String`s, and `ApprovalPreview` (which itself can hold `Vec<PathBuf>`). This forces every `RuntimeEvent` to be sized for the fattest variant. The `#[allow]` silences the lint instead of fixing it.

**Fix.** `ApprovalRequested(Box<ApprovalRequest>)`. Removes the lint, halves the in-memory size of every other event variant.

#### A.4 Missing event variants vs. the 00B plan

Plan 00B explicitly listed near-term variants that are **not** implemented yet:

```text
PlanReady
PatchPreviewReady
PatchApplied
EvidenceReady
WorkflowStepStarted
WorkflowStepFinished
SessionEnded
ApprovalCancelled / ApprovalTimeout
```

`SessionEnded` is the most painful gap: there's `TaskCompleted/Failed/Cancelled` but no symmetric "session is gone" event. Consumers that track per-session state never get a clean end signal.

**Fix.** Add at least `SessionEnded { session_id: SessionId, status: RuntimeSessionStatus }` now. The workflow/patch/plan variants can wait for the relevant capabilities.

#### A.5 `ApprovalDecision` only has `Approved`/`Rejected`

Real flows surface `Cancelled` (operator dismissed the modal), `Timeout` (no decision in time), `Skipped` (autonomy mode demoted while pending). Today exec mode emits `ApprovalRequested + TaskFailed { code: "approval_required" }` because there's no `Cancelled` decision to send.

**Fix.** Add `ApprovalDecision::Cancelled` and `ApprovalDecision::Timeout`. Update `ApprovalResolved::reason` semantics to be required for non-Approved.

#### A.6 `RuntimeError::owner` and `code` are stringly typed

```rust
pub struct RuntimeError {
    pub code: String,
    pub message: String,
    pub recovery_hint: Option<String>,
    pub retry_safe: bool,
    pub owner: String,
}
```

The codebase already uses `"approval_required"`, `"budget_limited"`, `"vac-exec"` as string constants. Drift between adapters is a question of when, not if.

**Fix.** Introduce an enum:

```rust
pub enum RuntimeErrorOwner { Core, Tui, Exec, Workflow, Operator, External }
pub enum RuntimeErrorCode { ApprovalRequired, BudgetLimited, ProviderUnavailable, ToolExecutionFailed, ValidationFailed, Unknown(String) }
```

The `Unknown(String)` arm preserves backward compat for unforeseen codes.

#### A.7 `RuntimeSession::created_at` uses non-deterministic `Utc::now()`

```rust
impl RuntimeSession {
    pub fn new(cwd: PathBuf, entrypoint: RuntimeEntrypoint, autonomy_mode: AutonomyMode) -> Self {
        Self { id: SessionId::new(), created_at: Utc::now(), /* ... */ }
    }
}
```

Tests constructing a `RuntimeSession` either side-step `new()` (the existing tests do) or accept timestamp drift. There's no clock injection.

**Fix.** Add `RuntimeSession::with_clock(clock: &dyn Clock, ...)` or accept `created_at: DateTime<Utc>` as a parameter. Keep `new()` as a convenience that calls `Utc::now()`.

#### A.8 Other smells

- `WorkflowInput = String;` — type alias loses room to grow. A struct with a single `text` field is a trivial refactor that doesn't break wire format if you serde-flatten.
- `PreviewKind` enum is parallel to `ApprovalPreview` (which can already report its kind). The kind enum exists only for serializing the kind separately — not actually done anywhere. Either delete `PreviewKind` or add the use case.
- `ApprovalResource` has no `Display`. Callers must `{:?}` it. Add `Display`.
- `RuntimeEvent` lacks `From<TaskStarted>` / `From<TaskCompleted>` / etc. impls. Adding them removes verbose `RuntimeEvent::TaskStarted(TaskStarted::new(task))` patterns. ~12 trivial impls.
- Tests don't include a serde round-trip. For a type intended as a contract, the wire shape should be pinned with `serde_json::to_string` / `from_str` round-trips and a couple of golden snapshots (`insta` is already a dev-dep).
- No `Ord/PartialOrd` on `RiskLevel`. Comparing risk levels (`if request.risk >= RiskLevel::Destructive { ... }`) is a natural operation.
- `tests.rs` test names are very long (`local_runtime_start_task_command_builds_expected_payload`). Tests inside a `local_runtime` module don't need the `local_runtime_` prefix.

## B. `vac-rs/exec/src/local_runtime.rs` — Exec adapter

### Strengths

- **Single projection point.** `LocalRuntimeBridge::map_core_event` is the *only* place that translates `vac_protocol::Event` to `RuntimeEvent`. Easy to audit.
- **Output trait abstracts JSON vs human modes.** `LocalRuntimeOutput` has clear methods (`print_config_summary`, `process_runtime_event`, `process_warning`, `print_final_output`).
- **Validation detection is folded into the same loop.** `is_validation_command(&begin.command)` decides whether to emit `ValidationStarted/Finished` alongside the regular `ToolCallStarted/Finished`. Good cohesion.
- **Approval failure path is explicit.** Emitting `ApprovalRequested + TaskFailed` together makes the headless behaviour deterministic.
- **Ctrl-C is wired to `Op::Interrupt`.** Spawned task aborts on shutdown.

### Issues

#### B.1 Unbounded buffers

```rust
struct LocalRuntimeBridge {
    /* ... */
    assistant_buffer: String,
    evidence: Vec<String>,
    final_message: Option<String>,
    validation_calls: HashMap<String, String>,
}
```

- `assistant_buffer.push_str(&delta.text)` on every `AgentMessageContentDelta` and `PlanDelta`. For a 50 KB final response this is O(n²) due to repeated reallocation, and the buffer is never freed mid-task.
- `evidence.push(format!("exec: {command_display} -> {}", exec_status_label(&status)))` is appended for every exec/mcp/web/image/patch tool call. A long-running session can accumulate thousands.

**Fix.** Either:

- Bound both with a soft cap (e.g. 256 KB for buffer, 1024 entries for evidence) and add `evidence_overflowed: usize` counter, or
- Stream `assistant_buffer` into the processor immediately and drop it. The buffer's only consumer is `final_message` fallback at `TurnComplete` — keep just the tail (last 4 KB).

#### B.2 Stale-turn events are silently dropped

```rust
if let Some(turn_id) = event_turn_id(&event.msg)
    && let Some(active_turn_id) = self.current_turn_id.as_deref()
    && active_turn_id != turn_id
{
    return BridgeOutput { events: Vec::new(), error_seen: false, terminate: false };
}
```

Silent drop is fine in steady state (a stale event from an older turn shouldn't leak), but it's untraceable: no `debug!` log, no counter. If the runtime ever projects a wrong turn id, this hides the bug.

**Fix.** Add `tracing::debug!(?turn_id, ?active_turn_id, "dropping stale turn event")` and consider a `dropped_events_for_other_turn: u64` field for `TaskCompleted` evidence.

#### B.3 Direct field mutation of `RuntimeTask`

See A.2 above. The exec bridge writes `task.id = self.task_id; task.status = RuntimeTaskStatus::Running;` after `RuntimeTask::new()` re-mints both. Per-event waste of `Uuid::new_v4()` and `RuntimeTaskStatus::Pending`.

**Fix.** Add `RuntimeTask::with_id_running(session_id, task_id, kind, prompt)` constructor.

#### B.4 Hardcoded approval-failure messages

Four near-identical functions (`runtime_approval_events_for_exec_approval`, `runtime_approval_events_for_patch`, `runtime_approval_events_for_permissions`, `runtime_approval_events_for_elicitation`) all build a `RuntimeError::new("approval_required", "... in exec mode", Some("run with bypass approvals only ..."), false, "vac-exec")`. Copy-paste with only the kind word changed.

**Fix.** Single helper `task_failed_due_to_required_approval(task_id, kind: &'static str) -> TaskFailed` plus a `RuntimeError::approval_required(action: ApprovalAction)` constructor in `vac-core`.

#### B.5 `LocalRuntimeHumanOutput` writes directly to `eprintln!`

```rust
impl LocalRuntimeOutput for LocalRuntimeHumanOutput {
    fn print_config_summary(&mut self, ...) { eprintln!("Vastar VAC v{} (local runtime)", ...); /* many more eprintln!s */ }
    fn process_runtime_event(&mut self, event: RuntimeEvent) -> RuntimeEventDisposition {
        match event { RuntimeEvent::AssistantDelta(delta) => { eprint!("{}", delta.text); /* ... */ } /* ... */ }
    }
}
```

Makes unit-testing the human output rendering essentially impossible without a subprocess. The structure and content of "vac exec" output is a UX-critical surface.

**Fix.** Inject `Box<dyn Write>` for stderr and `Box<dyn Write>` for stdout (the `final_message` print). Default to `std::io::stderr()`/`std::io::stdout()`. Tests can pass `Vec<u8>` and assert.

#### B.6 Magic tool-name strings

`"exec"`, `"apply_patch"`, `"web_search"`, `"image_generation"` appear as inline literals in `ToolCallStarted::new(...)`. If the contract ever wants to filter by tool kind, callers must string-match.

**Fix.** `pub enum BuiltinToolName { Exec, ApplyPatch, WebSearch, ImageGeneration, Mcp { server: String, tool: String } }` in `vac-core::local_runtime`. Keep `tool_name: String` on the event for forward compat, but provide a `From<BuiltinToolName> for String` to enforce the canonical labels.

#### B.7 `unreachable!` instead of structured failure

```rust
let RuntimeCommand::StartTask(start_task) = start_command else {
    unreachable!("runtime command start_task helper must return StartTask");
};
```

True today, but a panic is the wrong tool for an internal invariant. A `match` with the other arms returning `RuntimeError` is more honest.

#### B.8 Two helper duplication: `is_validation_command`

Not visible in the read but referenced. If it's a substring match like `command.contains("cargo test")`, that's fragile. Verify it consults a config-driven list (e.g. `config.validation_commands: Vec<String>`) or tool metadata.

## C. `vac-rs/exec/src/lib.rs` — Exec entry

### Strengths

- **Default prompt branch returns early into the runtime path.** `if command.is_none() { return run_local_runtime_prompt(...).await; }` is clean.
- **`log_local_runtime_start_task` documents intent in tracing.** Even when the legacy resume/review path runs, the runtime command is logged, making operator debugging consistent.

### Issues

#### C.1 `log_local_runtime_start_task` is misleading on resume/review

Resume builds a `RuntimeCommand::StartTask` only to log it, then proceeds via the legacy `InProcessAppServerClient`. The log line implies runtime-first behaviour where there is none. A reader looking at logs would conclude the system is further along than it is.

**Fix.** Either (a) only log when the path is actually runtime-first, or (b) annotate the legacy log with a `transport: "legacy"` field so it is unambiguous.

#### C.2 Heavy bootstrap is wasted on the runtime-first path

`run_main` builds the full `InProcessClientStartArgs` (cli_overrides, loader_overrides, cloud_requirements, feedback, log_db, environment_manager, config_warnings, session_source, enable_vac_api_key_env, client_name, client_version, experimental_api, opt_out_notification_methods, channel_capacity) before calling `run_exec_session`. The runtime-first path uses only `environment_manager`. Everything else is allocated and discarded.

**Fix.** Split `bootstrap_for_runtime_first(arg0_paths, &config) -> RuntimeBootstrap` (returns just the `Arc<EnvironmentManager>`) from `bootstrap_for_legacy(...) -> InProcessClientStartArgs`. Decide which to build based on `command.is_none()` *before* paying the cost.

#### C.3 `RequestIdSequencer` is reachable from runtime-first code path

It is constructed in `run_exec_session` regardless of branch. Cheap, but symptomatic of the bootstrap issue above.

#### C.4 Resume/review duplicate the `log_local_runtime_start_task` helper twice

The `Resume` arm and the `None` arm each call `log_local_runtime_start_task(&prompt_text, default_cwd.as_path())`. Identical args. Should be a single call after the match.

## D. `vac-rs/tui/src/local_runtime.rs` — TUI adapter

### Strengths (from the head + grep references)

- **Comments explicitly call out the runtime-first contract.** “Canonical input is `RuntimeCommand::StartTask`. The legacy `AppCommand::UserTurn` op the chatwidget submit path eventually builds is documented as a temporary compatibility transport behind `LegacyCompatTransport`. Step 00E retires the legacy transport.” Future readers will not be confused.
- **`RuntimeSubmitPlan::from_runtime_command` rejects non-`StartTask`.** Returning `Option<Self>` instead of silently falling back is the right call.
- **Tests assert canonical-input invariants** (`runtime_submission_plan_uses_start_task_as_canonical_input`, `runtime_submission_plan_rejects_non_start_task_commands`, `runtime_submission_plan_marks_legacy_userturn_as_compat_transport`).
- **Two-trace tracing on submit** (`vac::local_runtime::submit`) gives operators provable runtime-first engagement in `RUST_LOG=info` logs.

### Issues

#### D.1 113 KB / >2500 LoC in one file

`vac-rs/tui/src/local_runtime.rs` is the third-largest file in the TUI tree (after `chatwidget.rs` and `chat_composer.rs`). Mixed responsibilities visible from grep:

- Submit plan + canonical-input minting (`mint_start_task`, `RuntimeSubmitPlan`, `LegacyCompatTransport`, `start_trace`).
- Bridge (`RuntimeBridge`, projection helpers, approval-decision recording).
- Activity model (`RuntimeActivityState`, `RuntimeActivityItem`, `RuntimeActivityKind`, `RuntimeActivityStatus`, `RUNTIME_ACTIVITY_DEFAULT_LIMIT`).
- Labels (`label_for`, `activity_label_for`).
- Validation status mapping (`validation_status_from_exec_status`).
- Event helpers (`turn_abort_reason_label`).

**Fix.** Split into a directory module:

```
vac-rs/tui/src/local_runtime/
    mod.rs               // re-exports
    submit.rs            // RuntimeSubmitPlan, mint_start_task, LegacyCompatTransport, start_trace
    bridge.rs            // RuntimeBridge + projection
    activity.rs          // RuntimeActivityState, item, kind, status, RUNTIME_ACTIVITY_DEFAULT_LIMIT
    labels.rs            // label_for, activity_label_for
    approval.rs          // approval-decision projection
    legacy_compat.rs     // LegacyCompatTransport (or fold into submit.rs)
```

Greps and reviews become 10× cheaper. Each submodule independently testable.

#### D.2 Six dead-code warnings in CI today

From the `cargo check` output:

```
method `runtime_activity_mut` is never used
methods `prompt` and `cwd` are never used (RuntimeSubmitPlan)
methods `session_id`, `task_id`, `record_approval_decision`, `record_call_approval_decision`, `label_for`, `activity_label_for` are never used (RuntimeBridge)
function `validation_status_from_exec_status` is never used
function `turn_abort_reason_label` is never used
method `set_expanded` is never used (RuntimeActivityState)
```

These fall into two categories:

- **Scaffolding for upcoming sub-tasks** (`/activity` slash command implementation, expand toggle, validation projection through PTY-visible labels). Mark each with `#[allow(dead_code)]` plus a one-line comment naming the follow-up sub-task.
- **Truly dead** (unlikely all six). For each, decide: wire it up now or delete.

The current state is the worst of both worlds: warnings that everyone will start ignoring, hiding real future regressions.

#### D.3 `LegacyCompatTransport` is a single-variant enum

```rust
pub(crate) enum LegacyCompatTransport { UserTurn }
```

Guarded by `debug_assert_eq!(plan.legacy_compat_transport, LegacyCompatTransport::UserTurn)` at the chatwidget submission site. A single-variant enum offers exhaustiveness checking but no actual variation. If 00E retires the marker entirely, this is fine; if not, document the planned variants and add `#[non_exhaustive]`.

**Fix.** Either:

- Replace with a unit struct (`pub(crate) struct LegacyUserTurnTransport;`) + `PhantomData`-style marker; the debug_assert can become a type-system check, or
- Keep the enum and add the future variants now (e.g. `RuntimeDirect` for the post-00E path).

#### D.4 Field visibility in `RuntimeSubmitPlan` is over-exposed

All four fields are `pub(crate)`. The only consumer is `chatwidget.rs::submit_user_message_with_history_and_shell_escape_policy`. Narrower visibility (private fields + `pub(crate) fn start_task(&self)` etc.) means the chatwidget can't accidentally tamper with the plan.

## E. `vac-rs/tui/src/activity_sidebar.rs` — Sidebar UI

### Strengths

- **Pure rendering layer.** The sidebar reads `RuntimeActivityState` by reference; no state ownership.
- **Width-aware rendering.** `truncate_to_width` operates on `chars()` rather than bytes — multibyte safe.
- **Layout primitives are isolated.** `split_for_activity_sidebar` and the three constants (`MIN_TOTAL_WIDTH=120`, `WIDTH=36`, `HEIGHT=12`) are documented and reused.
- **Tests cover the boundary cases.** Narrow terminal returns `None`, wide terminal returns a sidebar inside bounds, truncate keeps short text intact, truncate appends ellipsis, full timeline strings, hidden-when-collapsed, all-items-when-open.
- **Constructor docstring** explicitly states the renderer does not own state — prevents future regressions.

### Issues

#### E.1 `truncate_to_width` is char-aware, not display-width aware

For East Asian / emoji content, `chars().count()` underreports the real terminal width. 你好 takes 4 columns but counts as 2 chars. The sidebar can therefore visually overflow on CJK content.

**Fix.** Use `unicode-width` (already a workspace dep). Replace `text.chars().count()` with `unicode_width::UnicodeWidthStr::width(text)` and walk graphemes (or grapheme clusters) for the truncation cursor.

#### E.2 `main_width_for_activity_sidebar` is a no-op

```rust
pub(crate) fn main_width_for_activity_sidebar(total_width: u16) -> u16 {
    total_width
}
```

Returns its argument unchanged. The doc comment justifies it as "self-documenting". This is the wrong tool: a comment at the call site does the job without an extra function call. If the function is reserved for future when sidebar reduces main width, the doc should say so explicitly and a TODO marker should be added.

**Fix.** Inline at call sites, or add `// TODO(00D-9): when sidebar becomes a docked column, return total_width - sidebar_width` and keep the function.

#### E.3 `RuntimeActivityStatus::Info => "-"` is a regular dash

Looks like a regular bullet line, not a status marker. A middle dot (`·`) or em-dash (`—`) reads better in context. Cosmetic.

#### E.4 `render` is a no-op when collapsed

```rust
if !self.state.is_expanded() || area.width == 0 || area.height == 0 { return; }
```

The test `sidebar_hidden_when_collapsed` asserts that the buffer cells stay empty. This relies on the parent painting an empty buffer first. If the parent ever ends up with stale cells in the sidebar region (e.g. partial repaint after a window resize), the previous frame's content will ghost.

**Fix.** Always clear the sidebar area at the start of `render` (`for y in area.y..area.y + area.height { for x in area.x..area.x + area.width { buf[(x, y)].set_symbol(" "); } }`). Costs ~432 cells of write; trivial.

#### E.5 `ActivitySidebar` doesn't reflect its role in the name

It's a renderer that holds a borrow, not the panel itself. `ActivitySidebarRenderer` is the more accurate name.

#### E.6 `lines.push(Line::from(""))` for spacers

This works but creates a `Line` containing an empty `Span`. `Line::default()` is more idiomatic and slightly cheaper.

## F. Architectural concerns (cross-cutting)

### F.1 Three modules named `local_runtime`, three different roles

- `vac_core::local_runtime` — contract types.
- `vac_exec::local_runtime` — protocol→contract projector + headless output trait.
- `vac_tui::local_runtime` — protocol→contract projector + activity state + canonical-input plan.

When importing a `RuntimeEvent` you must always say `vac_core::local_runtime::RuntimeEvent` to disambiguate. When jumping to `local_runtime` in your editor you get a 50/50 chance of the wrong module.

**Fix.** Rename the adapter modules: `vac_exec::runtime_adapter` and `vac_tui::runtime_adapter`. Keep `vac_core::local_runtime` as the canonical name.

### F.2 `LocalRuntimeBridge` (exec) and `RuntimeBridge` (tui) are duplicate projectors

Both translate `vac_protocol::Event` to `vac_core::local_runtime::RuntimeEvent`. Different files, similar logic (`ExecCommandBegin→ToolCallStarted`, `ExecCommandEnd→ToolCallFinished`, `ExecApprovalRequest→ApprovalRequested`, etc.). Drift is inevitable: a fix in one will not propagate.

**Fix.** Lift the projection into `vac_core::local_runtime::projection`:

```rust
pub trait RuntimeEventSink {
    fn push(&mut self, event: RuntimeEvent);
    fn warn(&mut self, message: String);
}

pub fn project_protocol_event<S: RuntimeEventSink>(
    state: &mut ProjectorState,
    event: vac_protocol::Event,
    sink: &mut S,
) -> ProjectorOutcome { /* ... shared logic ... */ }
```

Then exec and tui only own their own `RuntimeEventSink` impl + UI/output binding. The activity-state mapping in tui can live above the projection.

### F.3 No event ordering or sequence number

`RuntimeEvent` variants carry no sequence id. Replay, multi-stream merging, and observability all want monotonic ordering. Today the order is implicitly the producer's order; if events are ever buffered (e.g. into a vec for a frame), reordering can happen.

**Fix.** Either:

- Wrap events: `pub struct RuntimeEventEnvelope { pub seq: u64, pub session_id: SessionId, pub event: RuntimeEvent }`.
- Or add `RuntimeEvent::seq()` method backed by a per-variant field (more intrusive).

### F.4 Sessions/tasks are passed by value into events

`SessionStarted { session: RuntimeSession }` and `TaskStarted { task: RuntimeTask }` clone the full struct. For typical sessions this is fine, but for long-running ones with many turns the clones add up.

**Fix.** Carry `Arc<RuntimeSession>` for events, or split into `SessionStarted { session_id, cwd, entrypoint, autonomy_mode }` (denormalized) and let consumers query a session store for the rest.

### F.5 No "session engine" abstraction

Each adapter (exec, tui) tracks its own runtime state piecemeal. There is no central `LocalRuntimeSession` that owns `Vec<RuntimeTask>`, dispatches commands, and emits events. Plan 00F (proposed in `00D_RESIDUAL_LEGACY_AUDIT.md`) rightly flags introducing this trait. Confirming here that the absence is real and load-bearing for cleaner adapters.

### F.6 `AutonomyMode` is captured at session start, never mutated

No `RuntimeCommand::SetAutonomy { mode }`. Operators can't elevate autonomy mid-session. The contract baked in a static autonomy.

**Fix.** Add the command and a `SessionAutonomyChanged` event.

### F.7 No backpressure model

`Vec<RuntimeEvent>` returned from the bridge is processed synchronously. No channel, no flow control. Acceptable today but flagged for the multi-agent scenario in Tier 2.

## G. Testing gaps

### G.1 No serde round-trip tests

Nothing pins the wire format. If a contributor renames a field or changes serde attributes, no test breaks. The contract is meant to be **the** source of truth; that demands snapshot/round-trip tests.

**Fix.** Add `#[test]` per variant in `vac-rs/core/src/local_runtime/tests.rs`:

```rust
#[test]
fn task_started_round_trips_through_serde_json() {
    let event = RuntimeEvent::TaskStarted(TaskStarted::new(/* fixed task */));
    let json = serde_json::to_string(&event).unwrap();
    let parsed: RuntimeEvent = serde_json::from_str(&json).unwrap();
    assert_eq!(event, parsed);
}
```

For wire stability, add an `insta` snapshot of the JSON.

### G.2 Bridge filtering is untested

The stale-turn drop branch (B.2) has zero coverage. A bridge unit test that injects an event with the wrong turn id and asserts `events: vec![]` would lock in the contract.

### G.3 `LocalRuntimeHumanOutput` rendering is untested

Because it writes to `eprintln!` directly. After fixing B.5, add tests that run the processor over a fixture event stream and snapshot the captured output.

### G.4 No proptest on `truncate_to_width`

String algorithms with width awareness invite property tests:

- Output char count ≤ input width
- Output is a prefix of input (or input + ellipsis)
- Input shorter than width returns input unchanged

With `proptest` already common in Rust, this is ~20 lines.

### G.5 Activity state / sidebar interaction is partially tested

The sidebar tests cover hidden-when-collapsed and all-items-when-open. Missing:

- Re-pushing identical items — deduplication or accumulation?
- Ring-buffer cap (`RUNTIME_ACTIVITY_DEFAULT_LIMIT`) — does push past the cap drop oldest?
- Render after multiple expand/collapse cycles — no state leakage.
